# How Session Keys Work

## The core idea

A session key is a **temporary, limited-permission signing key** for a smart account. It's like a hotel key card — it opens your room for the duration of your stay, but it can't open other rooms, it can't access the safe, and the hotel can deactivate it instantly.

## The players

```
HUMAN (owner)                          AGENT (AI)
─────────────                          ──────────
Has: EOA private key                   Has: session private key
     (MetaMask, hardware wallet)            (generated by SDK)
Controls: the smart account            Controls: nothing
Can: do anything                       Can: only what the session allows
```

## What lives where

```
ON-CHAIN (Base)
├── Smart Account (Safe7579)
│   ├── Owned by: human's EOA
│   ├── Holds: USDC, ETH, tokens
│   └── Modules installed:
│       └── Smart Sessions validator
│           └── Session #1:
│               ├── Public key: 0xabc... (agent's session key)
│               ├── Allowed contracts: [Uniswap Router, USDC]
│               ├── Allowed functions: [exactInputSingle, approve]
│               ├── Spending limit: 100 USDC (cumulative on approve)
│               └── Expires: 2026-03-21 12:00 UTC

LOCAL (agent's machine)
├── .session.json
│   ├── sessionPrivateKey: 0xdef... (the signing key)
│   ├── permissionId: 0x123...
│   └── session: { ...full policy details }
```

## What happens when the agent sends a transaction

```
1. Agent wants to swap 10 USDC for ETH

2. SDK builds a UserOperation:
   - to: Uniswap Router
   - data: exactInputSingle(USDC, WETH, 10 USDC, ...)
   - signed by: session key (0xdef...)

3. Pre-flight check (off-chain, in the SDK):
   - Is Uniswap Router in the allowlist? → YES
   - Is exactInputSingle an allowed function? → YES
   - Is the value 0 (no raw ETH)? → YES
   → Passes

4. UserOp submitted to bundler (Pimlico)

5. Bundler submits to EntryPoint contract on Base

6. EntryPoint calls Smart Sessions validator on the Safe:
   - Is this session key registered? → YES
   - Is the session expired? → NO (still within 24h)
   - Is the target contract allowed? → YES (Uniswap Router)
   - Is the function selector allowed? → YES (exactInputSingle)
   → ALL CHECKS PASS → Execute the swap

7. Swap executes on Uniswap. Output tokens go to the smart account.
```

## What happens when the agent tries something not allowed

```
Agent tries to call transfer() on USDC (send tokens to external address):

Pre-flight check:
   - Is USDC in the allowlist? → YES
   - Is transfer() an allowed function? → NO (only approve() is allowed)
   → REJECTED before even submitting

Even if the pre-flight check had a bug:
   On-chain Smart Sessions validator:
   - Is transfer() allowed? → NO
   → UserOp REVERTS. No tokens moved. Agent wasted gas (sponsored by paymaster anyway).
```

## Revocation

The human runs `cd .agent-wallet && pnpm run revoke`. This sends a single transaction from the human's EOA to the smart account, which removes the session. The agent's session key becomes useless — it still exists locally, but the smart contract will reject every transaction signed with it.

The smart account address doesn't change. Funds don't move. The human can issue a new session key immediately if they want.
